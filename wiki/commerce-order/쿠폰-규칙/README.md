# 쿠폰 규칙

> 작성일: 2026-05-13 | 수정일: 2026-05-13 | 유형: 정책 | 관련 레포: example-order-api

## 개요

쿠폰의 종류, 발급·사용·중복 적용 규칙, 사용량 추적 방법을 정의합니다. 쿠폰은 마케팅·CRM이 발행하고 주문 도메인에서 검증·적용·회수합니다.

## 쿠폰 종류

| 종류 | 할인 형태 | 사용 예 |
|------|----------|--------|
| 정액 쿠폰 | 5,000원 할인 | 신규 가입 환영 쿠폰 |
| 정률 쿠폰 | 10% 할인 (상한 있음) | 시즌 프로모션 |
| 무료 배송 쿠폰 | 배송비 0원 | 일정 금액 이상 구매 시 |
| 카테고리 한정 쿠폰 | 특정 카테고리에만 적용 | "여성 의류 20% 할인" |
| 셀러 한정 쿠폰 | 특정 셀러 상품에만 | 셀러 자체 발행 쿠폰 |

## 쿠폰 검증 규칙

쿠폰을 적용하려고 할 때 다음을 순서대로 검증합니다:

1. **유효 기간**: 현재 시각이 `valid_from` ~ `valid_to` 사이
2. **사용자 보유**: 해당 사용자가 발급받은 쿠폰인지
3. **사용 횟수**: 이미 사용한 쿠폰이 아닌지
4. **최소 주문 금액**: 주문 금액이 `min_order_amount` 이상
5. **카테고리/셀러 한정**: 라인 아이템이 한정 조건을 충족
6. **사용량 한도**: 전체 발급 수량 한도(`global_usage_limit`)를 초과하지 않음
7. **중복 적용 가능 여부**: 다른 쿠폰과 함께 쓸 수 있는지

검증은 단계마다 fail-fast. 첫 실패에서 즉시 거부 이유를 반환합니다.

## 중복 적용 규칙

기본 정책은 **주문당 쿠폰 1개**입니다. 단, 다음은 동시 적용 가능:

- 정액·정률 쿠폰 1개 + 무료 배송 쿠폰 1개
- 정액·정률 쿠폰 1개 + 셀러 한정 쿠폰 1개 (다른 셀러 상품에 적용)

같은 종류의 쿠폰 중복 적용은 절대 불가. 정률 쿠폰 두 개를 동시에 적용하면 할인율이 비현실적으로 커지기 때문입니다.

## 사용량 카운터 (Redis)

쿠폰의 전체 사용량 한도(`global_usage_limit`)는 Redis `INCR` 카운터로 관리합니다:

```
INCR coupon:usage:{coupon_id}
  → 반환값이 global_usage_limit 초과면 → 사용 거부
```

원자성을 보장해 동시에 여러 사용자가 마지막 한 장의 쿠폰을 쓰려 할 때도 한 명만 성공합니다. 주문 실패 시(결제 실패 등) `DECR`로 카운터 복원.

## 쿠폰 사용 트랜잭션

쿠폰 적용은 주문 생성 트랜잭션의 일부로 처리됩니다:

```
[order-placement]
   │
   ├─ 1. 쿠폰 검증 (위 7단계)
   ├─ 2. Redis INCR (한도 확인)
   ├─ 3. order 레코드에 applied_coupon_id 기록
   ├─ 4. 사용자 보유 쿠폰을 used 상태로 전환
   │
   ├─ 결제 실패 시: 모두 롤백
   │   ├─ Redis DECR
   │   ├─ used → unused 복원
   │   └─ order 삭제
   │
   └─ 환불 시 (전체): 쿠폰 복원 여부는 정책에 따라
       ├─ 단순 변심: 복원 안 함 (1회 사용 처리)
       └─ 시스템 오류: 복원
```

## 회수 정책

환불 시 쿠폰을 사용자에게 돌려주는지는 환불 사유에 따라 다릅니다:

| 환불 사유 | 쿠폰 복원 |
|----------|----------|
| 단순 변심 | ❌ 복원 안 함 (사용 처리) |
| 상품 하자 | ✅ 복원 (사용자 잘못이 아님) |
| 배송 사고 | ✅ 복원 |
| 시스템 오류 | ✅ 복원 |

복원 시 쿠폰의 `valid_to`가 이미 지났으면 +7일 유예. 사용자에게 알림 발송.

## 의사결정 배경

- **왜 같은 종류 쿠폰 중복을 막는가**: 정률 쿠폰을 여러 개 적용하면 (1 - 0.1) × (1 - 0.1) = 0.81 처럼 비선형 누적이 됩니다. 사용자 기대(20% 할인)와 실제 동작(19%)이 다르고, 운영팀이 발행할 때마다 다른 쿠폰과의 상호작용을 검증해야 해서 운영 비용이 큽니다.
- **왜 Redis INCR로 한도 관리하는가**: MySQL 카운터로 하면 락 경합 때문에 인기 쿠폰 발급 시 응답이 느려집니다. Redis의 원자 연산이 동시성과 성능을 둘 다 만족합니다.
- **왜 단순 변심 환불에는 쿠폰을 복원하지 않는가**: 쿠폰은 사용자 행동을 유도하기 위한 인센티브입니다. 단순 변심은 사용자의 선택이므로, 쿠폰을 복원하면 쿠폰을 "리스크 없이 시도해보는 도구"로 오용될 수 있습니다.
