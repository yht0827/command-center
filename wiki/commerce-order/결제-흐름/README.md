# 결제 흐름

> 작성일: 2026-05-13 | 수정일: 2026-05-13 | 유형: 정책 | 관련 레포: example-order-api

## 개요

`order` 레코드가 `pending` 상태로 생성된 시점부터 PG를 통한 실제 결제 처리, 결과에 따른 상태 전이까지를 다룹니다. PG는 외부 시스템이고 응답 지연·실패가 빈번하므로 상태 머신과 복구 절차가 명확해야 합니다.

## 결제 상태 머신

```
            ┌──────────────────────────┐
            │                          │
[order: pending] ──► PG 호출 ──┬─► [payment: captured] ─► [order: paid]
                              │
                              ├─► [payment: failed]   ─► [order: payment_failed]
                              │                              │
                              │                              └─► 재고 점유 해제
                              │
                              └─► (응답 없음, 30초 타임아웃) ─► 비동기 콜백 대기
```

### 상태 정의

| 상태 | 의미 | 다음 가능 상태 |
|------|------|----------------|
| `pending` | PG 호출 직전 또는 응답 대기 | `captured`, `failed` |
| `captured` | 결제 성공, 금액 청구 완료 | `refunded`, `partial_refunded` |
| `failed` | 결제 실패 (영구) | (종료 상태) |
| `refunded` | 전체 환불 완료 | (종료 상태) |
| `partial_refunded` | 일부 환불 완료 | `refunded` (잔여도 환불 시) |

## PG 연동

결제 수단별로 다른 PG에 연결됩니다:

| 결제 수단 | PG | 특이사항 |
|-----------|----|----------|
| 신용카드 | 카드사 PG | 즉시 응답, 멱등 API |
| 계좌이체 | 은행 PG | 비동기 콜백 (1~수일 지연 가능) |
| 간편결제 (Pay) | 간편결제 PG | 외부 앱 전환 → 콜백 |
| 포인트 | 내부 시스템 | 외부 PG 호출 없음 |

PG 어댑터 레이어가 차이를 추상화합니다. 도메인 코드는 통합된 인터페이스(`pay`, `cancel`, `refund`)만 사용합니다.

## 멱등성

PG 호출은 멱등 키(`idempotency_key = order_id + retry_count`)를 사용합니다. 동일 키로 재호출하면 같은 결과를 반환하며 이중 청구가 발생하지 않습니다.

## 비동기 콜백

계좌이체·간편결제는 PG가 별도로 콜백(웹훅)을 보냅니다:

1. 결제 시도 시 `payment` 상태가 `pending`으로 유지됨
2. PG 콜백 수신 → `captured` 또는 `failed`로 전환
3. 콜백 검증: PG 서명 헤더로 위조 차단

콜백이 24시간 내 도착하지 않으면 운영 알림 발생, 운영자가 PG 콘솔에서 직접 결과 확인 후 수동 보정합니다.

## 실패 복구

PG 호출 자체가 실패(네트워크 오류, 5xx)한 경우:

- 자동 재시도 3회 (지수 백오프, 최대 30초)
- 모두 실패하면 `payment.failed` 처리
- 재고 점유 해제 이벤트 발행

PG 응답은 받았으나 우리 시스템 저장이 실패한 경우(드물지만 가능): 사용자 결제는 이미 진행됐는데 시스템상 `pending`인 상태. 정기 대사 배치(매 5분)가 PG 거래 내역과 우리 DB를 비교해 차이를 자동 보정하거나 알림을 발생시킵니다.

## SLO

| 지표 | 목표 |
|------|------|
| 결제 처리 응답 (카드, p95) | 3초 이내 |
| 결제 성공률 (PG 응답 기준) | 98% 이상 |
| 비동기 콜백 처리 지연 (p95) | 5초 이내 |
| 대사 배치 차이 발견 | 1건/일 미만 |

## 의사결정 배경

- **왜 PG 어댑터 레이어를 두는가**: PG마다 API 스펙·에러 코드가 다릅니다. 어댑터로 흡수해서 도메인 코드는 결제 수단을 신경 쓰지 않게 합니다. 새 PG를 추가할 때도 도메인 코드 변경 없이 어댑터만 추가하면 됩니다.
- **왜 멱등 키에 retry_count를 포함하는가**: 자동 재시도와 사용자 재시도를 구분하기 위해서입니다. 자동 재시도는 같은 결과를 기대하지만, 사용자가 직접 새로 시도하면 새 결제 시도로 처리됩니다.
- **왜 5분 주기 대사 배치를 두는가**: PG ↔ 우리 시스템 사이의 일시적 불일치는 PG 응답과 우리 트랜잭션 사이의 비동기 갭에서 발생합니다. 5분이면 그런 갭이 대부분 해소되어 자동 보정이 가능하고, 그래도 남은 차이는 사람이 확인할 만큼 작아집니다.
