# 가격 정책

> 작성일: 2026-05-13 | 수정일: 2026-05-13 | 유형: 정책 | 관련 레포: example-catalog-api

## 개요

한 상품에 동시에 적용될 수 있는 여러 가격(정가, 시즌 할인, 쿠폰 적용가)을 어떻게 보관하고, 어떤 순서로 노출 가격을 결정하는지를 정의합니다. 카탈로그의 가격은 노출용이며, 실제 결제 가격 계산은 주문 도메인(`commerce-order`)이 별도로 수행합니다.

## 가격의 종류

| 종류 | 보관 방식 | 용도 |
|------|-----------|------|
| **정가** (`base`) | 상품당 1건, 필수 | 셀러가 설정한 기준 가격 |
| **시즌 할인** (`seasonal`) | 0~N건, 기간 지정 | 봄세일, 블랙프라이데이 등 |
| **쿠폰 할인** (`coupon`) | 0~N건, 쿠폰 코드 연동 | 특정 사용자/조건에만 적용 |
| **MD 추천 특가** (`md_pick`) | 0~1건, 운영자 지정 | 검색 상위 노출용 |

## 노출 가격 결정 규칙

검색·상품 목록·상세 페이지에 노출되는 "표시 가격"은 다음 우선순위로 결정합니다:

1. **활성 시즌 할인 중 최저가** (현재 시간이 시작·종료 사이)
2. 활성 시즌 할인이 없으면 → **MD 추천 특가**
3. 둘 다 없으면 → **정가**

쿠폰 할인은 노출 가격에 반영하지 않습니다. 쿠폰은 사용자별 조건이라 카탈로그 노출에는 적합하지 않고, 주문 단계에서 적용됩니다.

## 가격 변경 흐름

`price-update-pipeline`이 처리합니다:

- **단건 변경**: `PATCH /products/{id}/prices/{price_id}` — 정가 인상/인하, 시즌 할인 등록·해제
- **일괄 변경**: `POST /price-batch` — 시즌 할인 전체 카테고리에 일괄 적용 (예: "여성 의류 카테고리 전체 20% 할인, 5/13~5/20")

가격 변경 직후 `search-index-sync`가 트리거되어 검색 인덱스의 `display_price`가 5초 내에 갱신됩니다.

## 가격 변동 이력

가격 변경 시 `price_history` 테이블에 변경 전/후 가격, 변경 사유, 변경자(시스템/운영자)를 기록합니다. 사용 목적:

1. 정산 정합성 (구매 시점의 가격과 현재 가격이 다를 때 추적)
2. 가격 조작 감지 (단기간 급격한 인상 후 다시 인하 → 할인율 부풀리기)
3. 셀러 분석 (가격 변경 빈도, 시즌 할인 효과 측정)

## 가격 이상치 검사

자동 검수 룰의 일부로 등록 시 가격을 검사합니다. 동일 카테고리의 중앙값 대비:

- **1/10 미만**: 오타·기재 실수 가능성 → 수동 검수 전환
- **10배 초과**: 가품·사기 우려 → 수동 검수 전환

이 임계값은 카테고리별로 다르며 [가격 이상치 룰 표](../가격-이상치-룰.md)에서 별도 관리합니다(아직 미작성).

## 의사결정 배경

- **왜 가격을 별도 entity로 분리하는가**: 한 상품이 여러 가격 정책을 동시에 가질 수 있고(정가 + 시즌 할인 + MD 특가), 가격은 상품 본문보다 훨씬 자주 변경됩니다. 본문 테이블에 가격을 넣으면 변경 빈도가 다른 데이터를 같은 행에 두는 비대칭이 생깁니다.
- **왜 쿠폰 가격은 카탈로그에 반영하지 않는가**: 쿠폰은 사용자별·조건별로 적용 가능 여부가 다릅니다. 카탈로그는 "이 상품을 누구에게 어떻게 노출할지"를 다루고, 사용자 조건은 주문 도메인에서 다루는 게 책임 분리 측면에서 적절합니다.
- **왜 노출 가격과 결제 가격을 분리하는가**: 카탈로그는 노출에만 책임지고, 실제 결제 금액은 주문 도메인이 쿠폰·포인트·배송비 등을 합산해 계산합니다. 두 가격이 동기화되어야 한다는 부담을 도메인 사이에 두지 않습니다.
