# 카테고리 구조

> 작성일: 2026-05-13 | 수정일: 2026-05-13 | 유형: 정책 | 관련 레포: example-catalog-api

## 개요

상품을 분류하는 카테고리 트리의 구조와 운영 규칙을 정의합니다. 카테고리는 (1) 셀러의 등록 시 분류 도구, (2) 구매자의 탐색 동선, (3) 검색 노출의 그룹핑 단위로 동시에 사용됩니다. 세 가지 책임이 한 트리에 묶이므로 깊이·다중 매핑·변경 영향을 신중히 다룹니다.

## 구조 규칙

| 규칙 | 값 |
|------|---|
| 최대 깊이 | 3단계 (예: "여성 > 의류 > 원피스") |
| 최대 자식 수 | 한 노드당 50개 |
| 상품 매핑 가능 노드 | 리프 노드만 |
| 한 상품의 카테고리 매핑 | 최대 2개 |
| 활성 카테고리 수 (전체) | 약 5,000개 운영 중 |

## 다중 매핑 규칙

한 상품은 최대 2개의 카테고리에 매핑할 수 있습니다. 사용 예시:

- 원피스가 "여성 > 의류 > 원피스" + "여성 > 신상품" 동시 분류
- 운동화가 "스포츠 > 신발" + "남성 > 신발" 동시 분류 (성별 카테고리 트리가 별도일 때)

**1차 카테고리**: 셀러가 등록 시 지정. 검색 인덱스의 `primary_category_path`에 저장.
**2차 카테고리** (선택): 운영팀이 큐레이션 차원에서 부여. 검색의 부가 시그널.

3개 이상 매핑은 허용하지 않습니다. 카테고리 의미가 흐려지고 검색 노출 우선순위가 복잡해지기 때문입니다.

## 카테고리 변경 영향

### 추가
- 새 리프 노드 추가는 안전 (기존 상품 영향 없음)
- 새 중간 노드 추가도 안전

### 이동/이름 변경
- 셀러·구매자 모두에게 보이는 텍스트가 바뀌므로 운영팀 사전 공지 필요
- 검색 인덱스의 `category_path` 재색인 필요 (해당 카테고리 하위 모든 상품)

### 삭제
- 상품이 매핑되어 있으면 차단
- 자식 노드가 있으면 차단
- 강제 삭제 시: 해당 카테고리 매핑 상품들이 자동으로 부모 카테고리로 이동 (단, 부모도 리프여야 → 일반적으로 불가능)

## 변경 시 재색인 절차

카테고리 트리 변경은 영향 범위가 크기 때문에 다음 절차를 따릅니다:

1. 변경안 운영팀 사전 검토 (영향 받는 상품 수 추산)
2. 변경 적용 시각 공지 (보통 트래픽 적은 새벽)
3. MySQL 카테고리 트리 변경
4. 영향 받는 상품들에 대해 `commerce.catalog.product.changed` 이벤트 일괄 발행 (검색 인덱스 재색인 유도)
5. 검색 인덱스 정합성 확인 (랜덤 샘플 50건 비교)

## 의사결정 배경

- **왜 깊이를 3단계로 제한하는가**: 깊이가 깊어질수록 셀러 등록 시 카테고리 찾기가 어려워지고, 구매자 탐색 동선도 길어집니다. 운영 데이터 분석상 3단계 이하에서 카테고리 매핑 정확도와 탐색 효율이 가장 높았습니다.
- **왜 상품을 리프 노드에만 매핑하는가**: 중간 노드(예: "여성 > 의류")에 직접 상품을 매핑하면, 그 노드의 하위 분류가 늘어났을 때 기존 상품들이 모두 미분류 상태가 됩니다. 리프 노드 강제는 분류 정확성을 시간이 지나도 유지하기 위한 장치입니다.
- **왜 다중 매핑을 2개까지만 허용하는가**: 1개로는 큐레이션 차원의 분류(신상품, 베스트셀러)를 별도로 표현하기 어렵습니다. 3개 이상은 의미가 흐려지고 검색 결과 중복 노출이 발생합니다. 2개가 절충점입니다.
