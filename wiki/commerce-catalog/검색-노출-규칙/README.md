# 검색 노출 규칙

> 작성일: 2026-05-13 | 수정일: 2026-05-13 | 유형: 정책 | 관련 레포: example-catalog-worker

## 개요

상품 카탈로그를 구매자에게 검색·탐색 가능하게 만드는 정책을 정의합니다. MySQL 본문을 Elasticsearch에 어떻게 정규화해 적재할지, 검색 결과 순서는 어떤 신호로 결정할지, 동기화 지연은 어디까지 허용할지를 다룹니다.

## 동기화 흐름

`search-index-sync`(컨슈머)가 `commerce.catalog.product.changed` 토픽을 구독합니다. 이벤트 수신 시:

1. `product_id`로 MySQL에서 최신 상품 본문 + 카테고리 경로 + 가격을 로드
2. 정규화 변환을 적용하여 검색용 도큐먼트 생성
3. Elasticsearch bulk API로 인덱싱
4. 실패 시 3회 재시도, 그래도 실패하면 DLQ 토픽으로 분기

본문은 이벤트에 실리지 않습니다 → [상품 등록 README](../상품-등록/README.md#의사결정-배경) 참조.

## 정규화 변환

| 원본 (MySQL) | 검색 문서 필드 | 변환 |
|---|---|---|
| `product.name` | `name` | 그대로 + `name.keyword`(정렬용) |
| `product.description` | `description_ko` | 한국어 형태소 분석기 (nori) 적용 |
| `category_ids[]` | `category_path[]` | 트리 경로 펼침 ("여성 > 의류 > 원피스") |
| `prices[]` | `display_price` | 우선순위 계산 후 최종 노출 가격 1개 |
| `images[]` | `thumbnail_url` | 대표 이미지의 CloudFront URL |
| 셀러 평점·재고 가용성 | `boost_signals.*` | 노출 우선순위 계산용 |

## 노출 우선순위

검색 결과 순서는 Elasticsearch의 `function_score`로 계산합니다:

```
점수 = (텍스트 매칭 BM25)
     + log(1 + 셀러 평점) × 0.5
     + (재고 가용성 ? 1.0 : 0.0)
     + (등록일 신선도 감쇠) × 0.3
     - (가격 이상치 페널티) × 0.2
```

- **재고 가용성**: 재고 0이면 1점 손실 (재고 도메인에서 별도 신호 제공)
- **신선도 감쇠**: 등록 후 30일 동안 가산점, 이후 0으로 수렴
- **가격 이상치**: 카테고리 중앙값에서 크게 벗어난 가격은 신뢰도 차원에서 감점

## SLO

| 지표 | 목표 |
|------|------|
| 등록 → 검색 노출 지연 (p95) | 5초 이내 |
| Elasticsearch 인덱스 가용성 | 99.9% |
| MySQL ↔ ES 데이터 정합성 | 자동 동기화로 0건 (수동 보정 없음) |

## 재시도/DLQ 정책

3회 재시도 후 실패하면 `commerce.catalog.product.changed.dlq` 토픽에 적재됩니다. DLQ에 메시지가 쌓이면 운영 알림이 발생하며, 운영자는 (1) 본문을 점검하고 (2) 수동 재처리 또는 폐기를 결정합니다.

## 의사결정 배경

- **왜 본문을 이벤트가 아닌 MySQL에서 로드하는가**: 이벤트는 작게 두고 본문은 단일 진실의 원천(MySQL)에서 가져옵니다. 본문 스키마가 바뀌어도 이벤트 스키마는 안정적입니다.
- **왜 카테고리 경로를 펼치는가**: 검색 쿼리에서 "여성 의류"로 검색하면 하위 노드 상품도 잡혀야 합니다. 트리를 매번 재귀 탐색하는 대신 인덱싱 시점에 펼쳐둡니다.
- **왜 재고 가용성이 검색 점수에 들어가는가**: 재고 0인 상품을 구매자에게 보여주는 것은 사용자 경험에 해롭습니다. 단, 완전 제외가 아니라 점수 감점만 하여 "다시 입고 예정" 같은 상품의 노출 가능성은 남깁니다.
