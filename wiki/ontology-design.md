# 온톨로지 설계 배경

`.claude/rules/ontology-rules.md`의 운영 규칙이 어떤 설계 결정 위에 만들어졌는지 정리한 문서입니다. 룰 파일에는 매 작업 시 참조되는 행동 규칙만 남기고, 1회성 배경과 의사결정 근거는 이 문서에서 다룹니다.

## 이것은 무엇인가

코드에 존재하는 비즈니스 개념과 그 관계를 기록하여, Claude가 코드를 빠르게 찾아가도록 돕는 **내비게이션 시스템**입니다.

각 entity는 `repo` + `package` 필드로 코드 위치가 고정되어 있습니다. Claude는 ontology를 읽고 "이 개념은 이 레포의 이 패키지에 있다"를 즉시 파악합니다. 이것이 ontology의 핵심 가치입니다.

## 이것이 아닌 것

- **디지털 트윈이 아닙니다.** 팔란티어 온톨로지처럼 비즈니스 운영 전체를 모델링하지 않습니다. 팔란티어는 Object Storage + OSDK라는 런타임 동기화 시스템으로 온톨로지와 데이터를 자동 연결합니다. Command Center에는 그런 자동화가 없습니다.
- **지식 그래프가 아닙니다.** RDF/OWL이나 Neo4j처럼 쿼리하는 데이터베이스가 아닙니다. YAML 파일을 Claude가 읽는 것이 전부입니다.
- **비즈니스 도메인 모델이 아닙니다.** 코드에 없는 개념(수동 워크플로, 조직 구조, 마케터의 엑셀 전달 과정 등)은 여기에 넣지 않습니다.

## 왜 코드 중심인가

Command Center 온톨로지는 사람과 Claude가 수동으로 유지보수합니다. 자동 동기화 파이프라인이 없으므로, **코드에서 존재를 확인할 수 있는 것만** 기록해야 정합성이 유지됩니다.

- `repo` + `package`가 가리키는 코드가 실존하는지는 검증 가능합니다.
- 코드가 변경되면 ontology도 갱신해야 하는데, 코드 앵커가 있어야 "이 entity가 아직 유효한가?"를 판단할 수 있습니다.
- 코드 앵커가 없는 entity는 시간이 지나면 현실과 괴리되고, 검증할 방법도 없습니다.

## 미채택 관계 유형

`ontology/tbox.yaml`의 `relation_types`는 10개입니다. 다음 유형들이 검토되었으나 채택되지 않았습니다:

`caches`, `fallback_to`, `mirrored_to`, `determines`, `transforms`, `bypasses`, `shared_schema`, `incentivizes`

표현이 어려운 관계가 반복되면 사용자에게 유형 추가를 제안하세요. 룰: `.claude/rules/ontology-rules.md § 관계 유형 확장`.
