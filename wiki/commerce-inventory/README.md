# 재고 관리 (commerce-inventory)

> 유형: 정책 | 관련 레포: example-inventory-api, example-inventory-worker

## 배경

상품의 실제 재고를 안전하게 유지하고, 주문 흐름과 동기화하는 도메인입니다. 카탈로그는 "이 상품이 있다"를 말하고, 주문은 "이 상품을 사겠다"를 말한다면, 재고 도메인은 "실제로 몇 개가 남아 있고, 누가 가져갔는지"를 추적합니다.

## 풀려는 문제

- **동시 구매 충돌**: 같은 시점에 여러 사용자가 마지막 한 개를 구매하면 한 명만 성공해야 합니다. 단순 차감으로는 race condition이 발생합니다.
- **결제 실패 시 재고 누락**: 주문 시 재고를 차감했는데 결제가 실패하면 그 재고는 영원히 묶입니다. 점유→확정 2단계 구조가 필요합니다.
- **장시간 결제 미완료**: 사용자가 결제 페이지를 열고 외출하는 경우, 그 시간 동안 재고가 묶이면 안 됩니다. TTL이 필요합니다.
- **재고 이력 추적**: 정합성 점검과 셀러 정산을 위해 모든 재고 변동의 사유·시각·처리자를 보관해야 합니다.
- **저재고 운영 알림**: 셀러가 재고 소진을 모르면 매출 기회를 놓치고 구매자에게 품절 경험을 만듭니다.

## 사용자

| 사용자 | 어떻게 사용하는가 |
|--------|--------------------|
| **구매자** | 직접 상호작용 없음. 가용 재고 > 0인 상품만 결제 가능 |
| **셀러** | 입고 처리, 재고 실사·조정, 저재고 알림 수신 |
| **타 도메인 (commerce-order)** | 점유·확정·복원 API 호출 (cross-domain) |
| **정산 시스템** | 재고 이동 이력으로 매출원가 계산 |

## 성공 기준

- 동시 구매 시 재고 초과 판매 **0건** (분산 락으로 보장)
- 결제 미완료로 인한 재고 누락 **0건** (TTL 자동 회수)
- 재고 정합성 (DB ↔ 이동 이력) **100%** (정기 감사 배치)
- 저재고 알림 발송까지의 지연 **30초 이내**
