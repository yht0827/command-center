# 행동 원칙

LLM의 흔한 코딩 실수를 줄이기 위한 가이드라인입니다.

### 1. 코딩 전에 생각하라

**추측하지 마라. 혼란을 숨기지 마라. 트레이드오프를 드러내라.**

구현하기 전에:
- **ontology 먼저**: 도메인 관련 요청이면 ontology를 먼저 읽어라(상세 절차: `.claude/rules/ontology-rules.md`).
- **가정**을 명시적으로 밝혀라. 확실하지 않으면 질문하라.
- 여러 해석이 가능하면 모두 제시하라 — 조용히 하나를 고르지 마라.
- 더 **단순한 접근법**이 있다면 말하라. 필요하면 반론을 제기하라.
- 뭔가 불명확하면 멈춰라. 무엇이 헷갈리는지 짚어라. 질문하라.
- **읽기 우선**: 수정 대상 코드를 먼저 읽어라. 읽지 않고 수정하지 마라. (도메인 관련이면 ontology → 코드 순서로 읽는다)
- **환각 금지**: 존재하지 않는 API, 패키지, 파일 경로, 설정 옵션을 지어내지 마라. 확실하지 않으면 먼저 확인하라.

### 2. 단순함 우선

**문제를 해결하는 최소한의 코드. 짐작으로 쓴 코드는 금지.**

- 요청받지 않은 기능을 추가하지 마라.
- 한 번만 쓰이는 코드에 **추상화**를 만들지 마라.
- 요청되지 않은 "유연성"이나 "설정 가능성"을 넣지 마라.
- 발생할 수 없는 시나리오에 대한 **에러 핸들링**을 하지 마라.
- 200줄로 썼는데 50줄로 가능하면 다시 작성하라.

자문하라: "시니어 엔지니어가 이거 **오버엔지니어링**이라고 할까?" 그렇다면 단순화하라.

### 3. 외과적 변경

**건드려야 할 것만 건드려라. 본인이 남긴 부산물만 정리하라.**

기존 코드를 수정할 때:
- 주변 코드, 주석, 포매팅을 "개선"하지 마라.
- 고장나지 않은 것을 **리팩토링**하지 마라.
- 다르게 할 수 있더라도 **기존 스타일**에 맞춰라.
- 관련 없는 **데드코드**를 발견하면 언급만 하라 — 삭제하지 마라.

본인의 변경으로 사용되지 않게 된 import/변수/함수는 제거하라.

검증 기준: **변경된 모든 라인은 사용자의 요청에 직접 연결**되어야 한다.

### 4. 목표 중심 실행

**성공 기준을 정의하라. 검증될 때까지 자가점검하며 반복하라.**

작업을 **검증 가능한 목표**로 변환하라:
- "유효성 검사 추가" → "잘못된 입력에 대한 테스트를 작성하고, 통과시켜라"
- "버그 수정" → "재현하는 테스트를 작성하고, 통과시켜라"
- "X 리팩토링" → "리팩토링 전후로 테스트가 통과하는지 확인하라"

다단계 작업에는 간략한 계획을 제시하라:
```
1. [단계] → 검증: [확인사항]
2. [단계] → 검증: [확인사항]
```

### 5. 변경 범위 최소화

**§4가 "무엇을 검증할 것인가"라면, §5는 "어떻게 쪼개서 실행할 것인가"다.**

- 여러 파일을 동시에 변경하기보다, 한 단위씩 변경하고 **중간 검증**을 수행하라.
- 뭔가 깨졌을 때 원인을 특정할 수 있도록 **변경 범위**를 최소화하라.
- 큰 변경을 한 커밋에 묶지 말고, 검증 가능한 단위로 분리하라.

### 6. 실패 대응

**에러가 나면 원인부터 파악하라. 같은 시도를 반복하지 마라.**

- 에러 발생 시 증상만 고치지 말고 **근본 원인**을 분석하라.
- 같은 명령을 재시도하기 전에 왜 실패했는지 먼저 이해하라.
- 방향이 틀렸으면 고치려 하지 말고 과감히 버리고 다시 작성하라. **매몰 비용**에 집착하지 마라.

### 7. AskUserQuestion 사용 규칙

**닫힌 질문에만 AskUserQuestion을 사용하라. 열린 질문에는 텍스트 출력 후 사용자 입력을 기다려라.**

- **닫힌 질문** (선택지가 명확할 때) → AskUserQuestion 사용
  - 예: "이어서 진행" / "새로 시작", "빌드 건너뛰기" / "직접 입력"
- **열린 질문** (자유 입력이 필요할 때) → 질문을 텍스트로 출력하고 사용자의 다음 메시지를 기다린다
  - 예: "이 도메인에서 풀려는 문제가 뭔가?", "관련 레포가 있는가?"

**빈 응답 처리:**
- AskUserQuestion에서 빈 응답(선택 없음, 텍스트 없음)을 받으면 **동의/승인/건너뛰기로 간주하지 마라.**
- 빈 응답 시 1회 재질문한다.
- 2회 연속 빈 응답이면 작업을 중단하고 사용자에게 상황을 설명한다.

### 8. 도구 사용 원칙

**스킬과 명령 형식에 대한 규칙. 이 워크스페이스의 도구 사용은 이 섹션이 SSOT다.**

**스킬 우선 사용**

- 커밋은 `/commit`, 워크트리는 `/worktree`, PR은 `/pull-request` 스킬을 사용하라. `git commit`, `git worktree`, `gh pr create` 등 git/gh 명령을 직접 실행하지 마라.
- 각 스킬은 이슈 키 파싱, 린트/테스트 pre-check, 안전 규칙 등을 내장한다 — 직접 실행 시 이 보호장치가 빠진다.
- 같은 유형 요청이 반복되면 기존 스킬 확장 또는 새 스킬 생성을 제안하라(§ 10. 피드백 반영).

**Bash 명령 canonical form** (permission 훅이 파싱 가능한 형태)

- **중첩 `()` 대신 순차 호출로 분리하라.** permission 훅이 서브셸 내부 명령을 파싱하지 못한다.
- **서브셸은 `(cd path && ...)` 패턴만 사용하라.** 훅이 자동 파싱하는 유일한 서브셸 패턴이다.
- **HEREDOC 중첩 금지** — `"$(cat <<'EOF'...)"` 같은 중첩은 zsh에서 파싱 에러를 일으킨다. 멀티라인 문자열은 `-m "..." -m "..."`처럼 분리하거나 별도 파일로 전달하라.
- **`grep` 패턴에 `\|`를 사용할 때** 인용문으로 감싸라: `grep "a\|b"`. 인용문 밖 `\|`는 파이프로 오인될 수 있다.

### 9. 폐기된 항목은 삭제

**진행하지 않기로 결정된 요구사항·작업 항목은 취소선(~~)으로 남기지 말고 행을 삭제하라.** 제거 사유는 커밋 메시지에 기록한다. status.md, PRD 등 모든 추적 문서에 동일하게 적용한다.

**발동**: 사용자가 항목 폐기를 결정한 시점에 즉시 적용. 추적 문서를 자동 스캔해서 정리하지는 않는다.

### 10. 피드백 반영

**사용자가 실수를 지적하면 기록하라. 작업 중 개선점을 발견하면 제안하라.**

- 같은 세션에서 지적받은 실수는 즉시 반영하고, 이후 동일 실수를 반복하지 마라.
- 사용자가 내 결과물을 직접 수정했다면, 수정 의도를 분석하여 컨벤션으로 제안하라.
- 코드 탐색 중 `wiki/`에 없는 도메인 지식을 발견하면 문서화를 제안하라.
- 같은 유형의 요청이 반복되면 skill 생성 또는 기존 skill 확장을 제안하라.
- 작업 흐름에서 수동/반복 작업을 발견하면 스크립트나 skill로 자동화를 제안하라.

발견 내용에 따라 적절한 위치에 반영을 제안하라:

| 발견 내용 | 반영 위치 |
|-----------|----------|
| 워크스페이스 전체 행동 원칙 | `.claude/rules/behavior.md` |
| 워크스페이스 진입 가이드(3계층 흐름·작업 범위) | `CLAUDE.md` |
| 특정 도메인 작업 룰 | `.claude/rules/{관련 파일}` |
| 비즈니스 정책/의사결정 | `wiki/{도메인}/` |
| 코드 구조/개념/관계 발견 | `ontology/abox/{도메인}.yaml` |
| 특정 프로젝트 코딩 규칙 | `projects/{name}/CLAUDE.md` |

제안 형식: `[자가학습] {반영 위치}에 추가 제안: 내용 — 근거`

**임의 반영 금지**: 사용자 확인을 받은 후 반영하라.

**적용 범위**: 이 §10은 *대화 중 새로 발견된* 컨벤션·문서화 후보·자동화 후보에 적용된다. 이미 합의된 작업의 일환으로 ontology/wiki를 함께 갱신하는 흐름(`.claude/rules/project-work.md` 개발 흐름 5단계, `.claude/rules/ontology-rules.md § 갱신 순서`)은 작업의 필수 산출물이므로 별도 승인이 필요하지 않다.

### 11. 서브에이전트 사용 기준

병렬 독립 작업이나 격리된 컨텍스트가 필요할 때 서브에이전트를 사용하라.
단일 파일 검색, 특정 파일 읽기, 순차 편집처럼 직접 도구 호출이 더 빠른 경우에는 직접 처리하라.
