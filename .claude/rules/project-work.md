# 프로젝트 작업 규칙

- 프로젝트 목록은 `.claude/workspace.json`에서 관리합니다.
- `/sync-projects`로 clone 및 최신화하세요. clone 시 격리 구조(`main/` + `worktrees/`)로 자동 설정됩니다.

## projects/ 작업 규칙

- 격리 구조 규칙은 `.claude/rules/workspace-structure.md`를 따릅니다.
- 코드 탐색 시 `projects/{name}/main/` 경로를 사용하세요. 워크트리가 있더라도 최신 코드 기준은 `main/`입니다.
- **세션 첫 탐색 시, 또는 최신성이 결과 정확성에 직접 영향을 줄 때**(API 시그니처·최신 동작 확인·머지 직후 재검증 등) 해당 프로젝트의 `main/`을 pull하여 최신화하세요. 단순 탐색이 반복될 때마다 pull하지 마세요.
- 코드 수정은 반드시 `/worktree create`로 워크트리를 만든 뒤 진행하세요.
- 해당 프로젝트 작업 시 `projects/{name}/main/CLAUDE.md`를 먼저 읽고 코드 컨벤션을 파악하세요.
- 코드 컨벤션을 발견하면 `projects/{name}/main/CLAUDE.md`에 반영을 제안하고, 격리 구조 루트에도 재복사합니다(`cp projects/{name}/main/CLAUDE.md projects/{name}/CLAUDE.md`). **이미 CLAUDE.md에 명시된 컨벤션은 재제안하지 마세요 — 같은 세션에서 동일 컨벤션 반복 제안 금지.**

## 개발 작업 흐름

코드 변경이 수반되는 요청을 받으면 아래 순서를 따른다. **코드부터 작성하고 문서를 나중에 맞추는 것은 금지.**

1. **ontology + wiki 확인**: 관련 도메인의 `ontology/abox/{도메인}.yaml`을 읽고, 영향받는 entity의 `wiki_doc`을 따라 정책 문서를 확인한다. 요청과 현재 설계/정책 사이의 이격을 파악한다.
   - **발동 예** (필수): PRD/요구사항 텍스트가 제시됨, 신규 기능 추가, 정책·스키마·계약 변경.
   - **Skip 예** (이 단계 생략 가능): 함수 내부 버그 수정, 타입 오류 수정, 로그 추가, 단순 리팩토링, 명백한 국소 변경.
   - 해당 도메인 ontology가 아직 없으면 `/new-domain`으로 먼저 생성한다.
2. **설계 반영**: 새 필드, 정책 변경, 스키마 변경 등이 있으면 wiki 문서를 먼저 갱신한다. 사용자와 설계 핑퐁이 필요하면 이 단계에서 수행한다. **문서를 수정하면 즉시 수정일을 갱신한다.**
3. **status.md 확인**: 구현할 FR/BR 항목을 `status.md`에서 확인하고, 변경 범위를 인지한다.
4. **코드 구현**: 설계 문서가 확정된 후에만 코드 작업에 진입한다.
5. **문서 마무리**: 구현 완료 후 `status.md` 상태 갱신(⬜→✅) + PR 설명 업데이트.
