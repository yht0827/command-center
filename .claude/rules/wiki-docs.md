# Wiki 문서 규칙

## wiki의 역할

wiki는 **사람이 읽는 비즈니스 정책 문서**다. LLM은 wiki에 직접 진입하지 않는다 — 진입 규칙의 SSOT는 `.claude/rules/ontology-rules.md § 탐색 흐름`.

| wiki에 담는 것 | wiki에 담지 않는 것 |
|----------------|---------------------|
| 비즈니스 정책/규칙 (Why, Rule) | API 스펙, DB 스키마 (→ 코드에서 직접 확인) |
| 의사결정 근거, 트레이드오프 | 소스 구조, 파일 목록 (→ ontology notes) |
| 사용자 시나리오, 흐름 (Flow) | 코드 레포 매핑 (→ ontology index.yaml repos) |
| 용어 해설 (Glossary) | 개념 간 관계/의존 (→ ontology relations) |
| 데이터 흐름 다이어그램 | |
| 구현 추적 (status.md) | |

> 3계층 역할 전체 정의: `.claude/rules/ontology-rules.md` 참조

## 도메인 디렉토리 구조

```
wiki/{도메인}/
├── README.md          ← 도메인 개요 (배경, 문제, 사용자, 성공기준)
├── glossary.md        ← 용어 사전 (사람이 이해할 수 있는 상세 설명)
├── architecture.md    ← 비즈니스 정책 인덱스 + 데이터 흐름 다이어그램
├── status.md          ← 구현 추적 (AC별 ✅/⬜)
│
├── {주제}/README.md   ← 주제별 비즈니스 정책/의사결정
└── ...
```

- 새 도메인 추가: `/new-domain` + ontology 도메인 파일 동시 생성
- `wiki/README.md`가 도메인 목록 인덱스 (GitHub 자동 렌더링)

## 문서 작성 규칙

- **architecture.md는 정책 인덱스**: 주제 문서 링크 + 데이터 흐름 다이어그램. 코드 구조/ERD는 ontology와 코드에 위임
- **주제 폴더 추가 시**: architecture.md 주제 테이블과 README.md 주제별 상세 테이블 모두 업데이트
- **주제 폴더명**: 주제명만 사용 (날짜 접두사 없음). 예: `팩토리-파이프라인/`
- **주제 문서 헤더**:
  ```
  > 작성일: YYYY-MM-DD | 수정일: YYYY-MM-DD | 유형: 정책 | 관련 레포: {레포명}
  > 관련 이슈: AFS-1, AFS-2
  ```
- **이슈 키는 헤더에만**: 본문에 인라인 삽입 금지
- **코드 반영 상태**: `status.md`에서 통합 추적
- **변경 이력**: git에 위임. 문서에는 현재 상태만 유지

## 최신화 의무

- 비즈니스 정책이 변경되면 해당 wiki 문서를 **반드시 업데이트**
- 코드 변경으로 인해 기존 정책 문서의 내용이 사실과 달라지면 수정 또는 "변경됨" 표시
- wiki 문서 추가/수정 시 대응하는 ontology entity의 `wiki_doc` 경로가 정확한지 확인
- 주기적으로 `/domain-audit`로 ontology ↔ wiki 정합성 점검

## 생애주기 규칙

문서는 생성만이 아니라 정리와 구조 점검도 필요하다.

- `/domain-cleanup`: 생애주기 정리 (stale, 완료 항목, 정합성)
- `/domain-audit`: 구조 점검 (중복, 겹침, ontology 정합성)

### 주제 유형

| 유형 | 설명 | 완료 후 처리 |
|------|------|-------------|
| **정책** | 서비스 운영 중 계속 참조되는 규칙/설계 (기본값) | 서비스 종료 전까지 유지 |
| **작업** | 특정 이슈 해결을 위한 일회성 설계/구현 기록 | 헤더에 `상태: 완료` 추가, 정리 후보로 감지 |

유형을 생략하면 **정책**으로 간주한다.

### status.md 정리

- 완료(✅)된 항목 중 **20일 이상 경과**한 항목은 정리 대상
- 정리 시 해당 행을 삭제 (git 이력에 보존)
- 진행 중(⬜) 항목만 남긴다
- **발동**: `/domain-cleanup` 실행 시 또는 사용자가 status.md 정리를 명시적으로 요청할 때. 일반 작업 중에는 자동 발동하지 않는다.

### 도메인 종료

서비스/기능이 종료된 도메인은 README.md 헤더에 상태를 기록:

```
> 상태: 종료 | 종료일: YYYY-MM-DD | 사유: {종료 사유}
```

`wiki/README.md` 인덱스에서도 활성/종료를 구분 표시. ontology index.yaml에서도 제거 또는 종료 표시.
