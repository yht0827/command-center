# Report Phase: 보고서 합성

Explore Phase의 전체 탐색 결과를 종합하여 최종 보고서를 생성한다. 오케스트레이터가 직접 수행한다.

**선행 로드**: Report Phase 진입 시 아래 2개를 Read한다.

라우팅 최적화에 의해 `report-guide.md`는 Explore Phase 마지막 메시지에서 `phase-report.md` Read와 **병렬 발행 가능**하다. 단, `SUMMARIES_FILE`은 Explore Phase에서 Write한 직후이므로 **별도 라운드**에서 Read한다 (Write/Read 경합 방지).

- `Read(SUMMARIES_FILE)` — Explore Phase에서 저장한 코드 스캔 결과. 컨텍스트에 Explore Phase 원본이 압축되었을 수 있으므로 반드시 파일에서 다시 로드한다. **Explore Phase Write 완료 후 읽을 것.**
- `Read(${CLAUDE_SKILL_DIR}/references/report-guide.md)` — 병렬 발행 가능

이후 5절, 7절에서 별도 Read 없이 메모리의 내용을 사용한다.

## 0. 단일 프로젝트 단축 경로

발견이 있는 프로젝트가 **1개**이면 다음 절을 건너뛴다:
- 3절 (프로젝트 관계 분석) — 비교 대상 없음
- 4절 (쿼리 초점 가중치) — 모든 발견을 핵심으로 취급
- 6절 (교차 분석) — 비교 대상 없음

1절 → 2절 → 5절 → 7절 → 8절 → 9절 순서로 진행한다.
5절에서 템플릿 선택과 합성은 동일하게 수행하되, 초점 분류 없이 전체를 주요 섹션으로 작성한다.
탐색 프로젝트 목록은 1줄로 축약하거나 생략한다.

## 1. 유사 기능 확인

Explore Phase에서 "유사 기능"으로 분류된 항목이 있으면, 보고서 작성 전에 사용자에게 포함 여부를 확인한다.

### 자동 승격

사용자에게 묻기 전에, 유사 기능 중 **다른 프로젝트에서 "발견된 구현"으로 이미 확인된 동일 정책**이 있는지 교차 대조한다.

자동 승격 조건 (모두 충족):
1. 유사 기능의 **정책명/키워드**가 다른 프로젝트의 "발견된 구현" 항목과 **동일하거나 밀접하게 대응**한다.
2. 유사 기능의 **비즈니스 역할**이 해당 "발견된 구현"과 **같은 도메인**에 속한다.

자동 승격된 항목은 "발견된 구현"으로 이동하고, 사용자 선택지에서 제외한다.

### 사용자 확인

자동 승격 후 남은 유사 기능이 있으면:

```
탐색 중 쿼리와 직접 관련은 아니지만 유사한 기능이 발견되었습니다.

  1. [프로젝트명] — [유사 기능명]: [무엇이 유사하고 무엇이 다른지]
  2. ...

보고서에 포함할 항목을 선택해주세요. (복수 선택 가능, 스킵하려면 "없음" 선택)
```

AskUserQuestion으로 multiSelect 선택지를 제시한다.
- 사용자가 선택한 항목은 "발견된 구현"에 합류시킨다.
- 선택하지 않은 항목은 제외한다.
- 자동 승격 + 남은 유사 기능 모두 0건이면 이 단계를 건너뛴다.

## 2. 전체 미발견 체크

모든 프로젝트에서 발견이 0건이면 (유사 기능 포함 후에도):
```
## 정책 탐지 보고서: <쿼리>

탐색한 <N>개 프로젝트에서 관련 정책 구현을 찾지 못했습니다.

가능한 원인:
- 해당 정책이 탐색 범위 밖에 구현되어 있을 수 있습니다.
- 코드에서 다른 용어를 사용하고 있을 수 있습니다.

다음을 시도해보세요:
- 다른 표현으로 쿼리: `/lens <다른 키워드>`
- 특정 프로젝트만 지정: `/lens <프로젝트명>에서 <쿼리>`
```
SESSION_DIR을 `realpath`로 정규화한 후, 비어있지 않고 `.lens/` 하위인지 확인한 후에만 `Bash(rm -rf <SESSION_DIR>)` — 세션 디렉토리 정리.
Report Phase 완료. 아래 단계를 건너뛴다.

## 3. 프로젝트 관계 분석

Explore Phase 결과에서 **프로젝트 간 호출 관계**를 추출한다.

### 호출 관계 식별

각 프로젝트의 발견에서 외부 API 호출 패턴을 찾는다:
- `*ApiClient`, `*Client`, `*Adapter`, `*Provider` 클래스가 다른 프로젝트의 API를 호출
- 호출 대상 프로젝트가 CANDIDATES에 포함되면 호출 관계 성립

### 탐색 프로젝트 목록 구성

발견이 있는 프로젝트를 불릿 리스트로 나열한다. 각 항목에 역할을 한 줄로 요약하고, 호출 관계가 있으면 자연스럽게 포함한다.

- **<프로젝트A>** — <역할>. <프로젝트B>, <프로젝트C>를 호출하여 데이터를 조합
- **<프로젝트B>** — <역할>
- **<프로젝트C>** — <역할>

관계 유형별 작성법:
- **호출 관계 있음**: "A가 B를 호출" / "B, C를 호출하여 데이터를 조합" 등으로 역할 설명에 포함
- **독립 구현**: 관계 언급 없이 역할만 기술
- **단일 프로젝트**: 목록을 1줄로 축약하거나 생략 가능
- 상세 모드(`DETAIL_MODE = true`)에서는 API 엔드포인트를 병기한다.

## 4. 쿼리 초점 가중치

보고서 합성 전에, 각 발견의 **쿼리 초점 관련도**를 분류하여 보고서 비중을 결정한다.

### 초점 판정

쿼리에서 **"무엇이 어떻게 되는가"**의 직접 대상을 식별한다:
- "홈피드 노출 정책" → 직접 대상 = **피드에 표시되는 상품**. 탭 구성은 상위 컨테이너.
- "장바구니 노출 정책" → 직접 대상 = **장바구니에 표시되는 상품/옵션**.
- "홈탭 구성 정책" → 직접 대상 = **탭 목록과 순서**.

### 관련도 분류

각 발견을 3단계로 분류한다:

| 등급 | 기준 | 보고서 비중 |
|------|------|-----------:|
| 핵심 | 쿼리의 직접 대상이 동작하는 조건/규칙 | 주요 섹션. 상세 기술 |
| 구조 | 직접 대상의 상위 컨테이너/프레임워크 | 보조 섹션. 요약 기술 |
| 운영 | 관리/설정/어드민 측 구현 | 탐색 프로젝트 역할에 포함하거나 생략 |

### 보고서 반영

- **핵심** 발견 → 정책 결과의 **주요 섹션**으로 상세 기술. 보고서 상단에 배치.
- **구조** 발견 → **보조 섹션**으로 축약하거나, 핵심 섹션의 전제 조건으로 간략 언급. 예: "탭 구성" 섹션을 핵심 "상품 노출" 섹션 뒤에 짧게 배치. 단, 구조 발견이 쿼리의 비즈니스 맥락에서 핵심과 동등하게 중요하면 주요 섹션에 포함한다.
- **운영** 발견 → 탐색 프로젝트 역할 설명에 포함하거나, "확인이 필요한 사항"에 기술.
- 핵심 발견이 0건이고 구조 발견만 있으면, 구조 발견을 주요 섹션으로 승격한다.

## 5. 정책 결과 합성

Explore Phase의 프로젝트별 발견 사항을 **"대상에게 무슨 일이 일어나는가"** 관점으로 재합성한다. 프로젝트 경계를 무시하고, 같은 정책 결과를 만드는 발견들을 하나의 항목으로 합친다. **4절에서 분류한 관련도에 따라 비중을 조절**한다.

템플릿 선택, 소그룹화 규칙, 합성 규칙의 상세는 `references/report-guide.md`를 참조한다 (Report Phase 진입 시 이미 Read 완료).

## 6. 교차 분석

**상세 모드에서만** 별도 섹션으로 포함한다.
**요약 모드에서는** "확인이 필요한 사항"에 차이점/갭을 통합한다.

발견이 있는 프로젝트가 2개 이상인 경우에만 수행한다.

### 분석 항목
- **공통 패턴**: 2개 이상 프로젝트에서 동일/유사한 정책.
- **차이점**: 같은 정책이지만 값이나 규칙이 다른 경우.
- **갭 식별**: 일부 프로젝트에만 존재하는 정책.

### 일관성 판정
| 판정 | 기준 |
|------|------|
| 일관적 | 관련 프로젝트 전체에서 동일한 정책/규칙/값이 확인됨 |
| 부분적 차이 | 동일 정책이 존재하나 값이나 세부 규칙이 프로젝트마다 다름 |
| 불일치 | 정책 자체가 상충하거나, 일부 프로젝트에 해당 정책이 아예 없음 |

## 7. 보고서 작성

보고서 공통 구조, 마크다운 활용 원칙, 모드별 작성 원칙의 상세는 `references/report-guide.md`를 참조한다 (Report Phase 진입 시 이미 Read 완료).

## 8. 보고서 출력

작성된 보고서를 **사용자에게 직접 출력**한다.

## 9. 영향도 분석 분기

### SUMMARIES_FILE 확인

`SUMMARIES_FILE`은 Explore Phase에서 이미 저장되었다. Report Phase 선행 로드에서 Read한 내용을 사용한다.

### --idea 플래그 확인

Prepare Phase 0절에서 파싱한 `IDEA_RAW`가 존재하면:
- `IDEA_CONTEXT = { idea: IDEA_RAW, clarifications: null }`
- `Read(${CLAUDE_SKILL_DIR}/phases/phase-impact.md)` 후 Impact Phase로 진행한다. (라우팅 최적화 미적용 — Impact Phase Task 프롬프트가 사용자 입력에 의존)

### 아이디어 질문

`IDEA_RAW`가 없으면 AskUserQuestion **1회**로 질문한다:

```
변경하려는 아이디어가 있으면 알려주세요. 영향도를 분석합니다.
```

- 선택지:
  - "분석 없이 마치겠습니다"
- 사용자가 "Other"로 아이디어를 직접 입력하면 → 아이디어를 입력받은 것으로 처리한다.

- **아이디어를 입력받으면**:
  - `IDEA_CONTEXT = { idea: <입력된 아이디어>, clarifications: null }`
  - IDEA가 불명확하면 **1회만** 보강 질문 (대상 범위/변경 규모/제약 조건, 최대 3개). 답변을 `clarifications`에 저장.
  - `Read(${CLAUDE_SKILL_DIR}/phases/phase-impact.md)` 후 Impact Phase로 진행한다. (라우팅 최적화 미적용 — Impact Phase Task 프롬프트가 사용자 입력에 의존)

- **"분석 없이 마치겠습니다"를 선택하면**:
  ```
  추가로 확인하고 싶은 정책이 있으면 `/lens <쿼리>`로 다시 실행해주세요.
  특정 프로젝트만 보려면 `/lens <프로젝트명>에서 <쿼리>`로 지정할 수 있습니다.
  ```
  요약 모드일 때 추가:
  ```
  더 자세한 보고서가 필요하면 `--detail` 옵션을 추가해주세요.
  ```
  SESSION_DIR을 `realpath`로 정규화한 후, 비어있지 않고 `.lens/` 하위인지 확인한 후에만 `Bash(rm -rf <SESSION_DIR>)` — 세션 디렉토리 정리.
  여기서 종료한다. Impact Phase, Impact-Report Phase는 실행하지 않는다.
