# 보고서 가이드

Report Phase 5절·7절에서 참조한다.

## 템플릿 선택

쿼리의 성격에 따라 가장 적합한 템플릿을 선택한다. 혼합도 가능하다.

- **A. 노출/필터링**: 제외 조건(불릿) + 표시 조건(표) + 사용자별 분기
- **B. 상태/흐름**: 상태 전이(표) + 각 상태의 조건(볼드 + 불릿)
- **C. 조건/규칙**: 허용 조건(불릿) + 차단 조건(불릿)

## 템플릿 A: 노출/필터링

쿼리에 "노출", "표시", "숨김", "필터" 등이 포함될 때 사용한다.

제외 조건을 판정 주체/시점별로 소그룹화(볼드 소제목 + 출처 병기) → 표시 조건(표) → 사용자별 분기(표, 해당 시).

```markdown
### <대상>에서 제외되는 조건

**<소그룹 제목>** — <출처 프로젝트>에서 판정

- **<조건명>** — <규칙을 자연어로>. `<핵심 값>`
- ...

### <대상>에 표시되는 조건

| 노출 규칙 | 내용 |
|----------|------|

### 사용자별 분기 (해당 시)

| 사용자 유형 | 경로 | 특이 조건 |
|------------|------|----------|
```

## 템플릿 B: 상태/흐름

쿼리에 "상태", "절차", "프로세스", "등록" 등이 포함될 때 사용한다.

상태 전이(표) → 각 상태의 진입 조건/핵심 값(볼드 + 불릿).

```markdown
### 상태 전이

| 현재 상태 | 전이 가능 상태 | 조건 |
|-----------|--------------|------|

### 각 상태의 조건

**<상태명>** — <출처>

- **진입 조건**: <자연어>
- **핵심 값**: `<값>`
```

## 템플릿 C: 조건/규칙

쿼리에 "정책", "규칙", "한도", "제한" 등이 포함될 때 사용한다.

허용 조건 + 차단 조건을 불릿으로 나열. 출처를 괄호에 병기.

```markdown
### 허용 조건

- **<조건명>** — <규칙>. `<핵심 값>` (<출처>)

### 차단 조건

- **<조건명>** — <규칙>. `<핵심 값>` (<출처>)
```

## 합성 규칙

1. **중복 제거**: 여러 프로젝트에서 동일한 정책이 발견되면 하나의 항목으로 합치고, 출처에 모든 프로젝트명을 나열한다. 예: "(<프로젝트A>, <프로젝트B>에서 확인)"
2. **추상화 수준 통일**: 구현 상세(함수명, 내부 로직)가 아닌 비즈니스 규칙 수준으로 서술한다. 예: "GetAllTacasService에서 filter == SELLABLE이면 winnerItem.stock.visibleOnPlp를 검사" → "래플 상품은 PLP에서 제외"
3. **인과 체인 압축**: "A가 B를 호출하고 B가 C를 판단한다"를 "C이면 제외 (출처: B)"로 압축한다.
4. **핵심 수치 분리**: 정책에서 발견된 상수/설정값은 정책 결과 섹션과 별도로 "핵심 수치" 코드블록에 모아 정리한다.
5. **요약 모드**: 주요 항목 위주로 제한. 항목이 많으면 핵심만 표시 + "외 N건".
6. **상세 모드**: 모든 항목 포함. 각 조건에 하위 불릿으로 상세 설명 추가. 출처에 파일명 병기.

## 보고서 구조

```markdown
## <쿼리>

> N개 프로젝트 감지 | M개 후보 탐색

---

### 탐색 프로젝트

- **<프로젝트>** — <역할 한 줄>. <다른 프로젝트를 호출하면 명시>

---

### [핵심 — 쿼리의 직접 대상에 대한 정책 결과]

(Report Phase 4절 "쿼리 초점 가중치"의 핵심 등급. 5절 템플릿 적용)

---

### [구조 — 상위 컨테이너/프레임워크 정책] (해당 시)

(Report Phase 4절 "구조" 등급. 핵심보다 축약)

---

### 핵심 수치 (상수/설정값이 있을 때)

 ```
 <항목명>    <값> (<변수명>)
 ```

---

### 확인이 필요한 사항 (차이/갭이 있을 때만)

> <내용>. <관련 프로젝트>.
```

## 작성 원칙

### 마크다운 활용

| 정보 성격 | 마크다운 요소 |
|----------|-------------|
| 조건/규칙 나열 | **불릿** + 볼드 조건명 + 인라인 코드 값 |
| 비교/대조 | **표** (2~3열). 4열 이상 피함 |
| 핵심 수치/상수 | **코드블록** (key-value 정렬) |
| 주의/확인 사항 | **blockquote** (`>`) |
| 문맥 설명 | **일반 텍스트**. 섹션 시작에 한 줄 요약 |
| 섹션 구분 | **수평선** (`---`) |

### 공통 원칙

- 비즈니스 용어 우선. 코드 용어는 인라인 코드(`` ` ``)로 괄호 안에 병기.
- 권장사항에는 코드 변경 제안을 넣지 않는다. "확인이 필요한 사항"만 안내한다.
- 탐색 실패 프로젝트는 blockquote로 별도 표기.
- **마크다운 다양성**: 불릿, 표, 코드블록, blockquote를 섞어 사용한다. 같은 형식이 3회 이상 연속 반복되지 않도록 한다.
- **수평선**: 주요 섹션(호출 흐름, 정책 결과, 핵심 수치, 확인 사항) 사이에 `---`를 삽입한다.

### 요약 모드 추가 원칙

- 전체 보고서가 **화면 2~3페이지 이내**.
- 출처는 소그룹 제목에 프로젝트명만 표기. 파일명 생략.
- `--detail` 안내를 완료 메시지에 포함.

### 상세 모드 추가 원칙

- 호출 트리에 API 엔드포인트 포함.
- 각 조건에 하위 불릿으로 상세 설명 추가.
- 출처에 파일명 병기: `프로젝트명 (파일명)`.
- 교차 분석 섹션 별도 포함.
