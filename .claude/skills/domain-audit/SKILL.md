`wiki/` 문서 간 의미적 관계를 분석하고 구조 개선을 제안하는 스킬.

`/domain-cleanup`이 "오래된 것 정리"라면, `/domain-audit`은 "구조가 맞는지 검토"다.

---

## 인자

- `ARGS` (선택): 비교할 주제/도메인명을 공백으로 구분.
  - 비교 모드: `/domain-audit 쿠폰 혜택` → 두 주제를 비교
  - 전체 스캔: `/domain-audit` → 전체 도메인/주제를 스캔하여 관계 분석

주제명은 유연하게 매칭한다:
- 정확한 폴더명: `팩토리-파이프라인`
- 도메인/주제: `asset-factory/팩토리-파이프라인`
- 부분 매칭: `파이프라인` → `팩토리-파이프라인` 매칭 시도

## 분석 항목

### 1. 내용 중복/겹침

- 비교 대상 문서를 읽고, 동일하거나 유사한 내용이 있는지 분석한다
- 같은 개념을 다른 용어로 설명하는 경우도 감지한다
- 겹침 정도를 판단한다: 완전 중복 / 부분 겹침 / 관련 있지만 다름

### 2. 통합/분리 제안

- **통합 후보**: 두 주제가 실질적으로 같은 것을 다루는 경우
- **분리 후보**: 한 주제가 너무 많은 범위를 다루어 분리가 필요한 경우
- **참조 관계**: 별개이지만 상호 참조가 필요한 경우

### 3. 도메인 경계 검토 (전체 스캔 시)

- A 도메인 문서에서 B 도메인 개념을 깊게 다루는 경우 감지
- 도메인 간 공유되는 개념이 있으면 `wiki/glossary.md`(공통 용어)로의 추출을 제안

### 4. ontology ↔ wiki 정합성

- `ontology/index.yaml`의 도메인 목록과 `wiki/` 디렉토리가 일치하는지 확인
- 각 ontology entity의 `wiki_doc`이 실존하는 파일을 가리키는지 확인
- ontology에는 있지만 wiki 문서가 없는 entity를 리포트 (문서 작성 필요 후보)
- wiki에는 있지만 ontology에 대응 entity가 없는 문서를 리포트 (ontology 갱신 필요 후보)

### 5. ontology axiom 위반 점검

`ontology/tbox.yaml`을 Read하여 axiom 규칙을 로드한 뒤, 각 도메인 abox 파일과 `abox/infra.yaml`을 순회하며 위반을 점검한다.

**점검 절차:**

1. **domain_range 위반**: 각 relation의 `from`/`to` entity의 `type`이 tbox `domain_range`에 정의된 허용 타입에 포함되는지 확인. `[all]`이면 모든 타입 허용.
2. **existence 위반**: 각 entity의 `type`에 따라 tbox `existence`에 정의된 필드 필수/금지 규칙을 확인. `required`인데 빠졌거나, `forbidden`인데 값이 있으면 위반.
3. **disjointness 위반**: 하나의 entity에 복수 type이 지정되었는지 확인.
4. **closure 위반**: entity type, relation type이 tbox `closure`에 정의된 허용 값 내인지 확인. `id_pattern` 정규식에 entity id가 매칭되는지 확인.
5. **structural 위반**: 어떤 relation에도 참여하지 않는 고아 entity가 있는지 확인. cross-domain의 from/to가 `{도메인id}/{entity-id}` 형식인지 확인.

**리포트 형식:**

```markdown
### axiom 위반
| 도메인 | entity/relation | 위반 유형 | 상세 |
|--------|----------------|----------|------|
| asset-factory | consumes: x→y | domain_range | from이 data인데 consumes는 process만 허용 |
```

## 실행 흐름

### 비교 모드

```
1. 인자에서 주제/도메인명 파싱 → 실제 경로 매칭
2. 대상 문서들을 읽고 내용 비교
3. 분석 결과 리포트 출력
4. 구조 변경이 필요하면 제안 (실행은 사용자 판단)
```

### 전체 스캔 모드

```
1. 모든 도메인의 주제 문서 목록 수집
2. 각 문서의 핵심 키워드/개념 추출
3. 문서 간 유사도 분석 (키워드 겹침 기반)
4. 겹침이 높은 쌍을 후보로 리포트
5. 도메인 경계 검토
```

## 리포트 형식

```markdown
# 도메인 구조 분석 리포트

## 비교: {주제A} vs {주제B}

### 겹침 분석
| 항목 | 주제A | 주제B | 판정 |
|------|-------|-------|------|
| 보상 정책 | 성장 보상 설명 | 미션 보상 설명 | 부분 겹침 |

### 제안
- **통합**: "보상 정책"을 별도 주제로 분리하여 양쪽에서 참조
- **참조 추가**: 주제A에서 주제B 링크 추가

---

## 전체 스캔 결과 (전체 스캔 시)

### 겹침 후보
| 주제 쌍 | 도메인 | 겹침 키워드 | 판정 |
|---------|--------|-----------|------|
| 성장-시스템 / 미션-시스템 | 고양이키우기 | 보상, 레벨 | 부분 겹침 |

### 도메인 경계 이슈
| 문서 | 참조하는 외부 도메인 | 제안 |
|------|-------------------|------|
| ... | ... | 공통 용어로 추출 |
```

## 제약

- **읽기 전용**: 이 스킬은 분석과 제안만 한다. 파일을 수정하지 않는다
- **판단은 제안**: "합쳐야 한다"가 아니라 "합치는 것을 검토해보라"는 톤
- 문서 수가 많을 때 전체 스캔은 시간이 걸릴 수 있다 — 진행 상황을 중간 출력한다
